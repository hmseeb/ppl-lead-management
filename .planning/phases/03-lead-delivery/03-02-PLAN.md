  <action>
Create a single migration file with two functions and two pg_cron schedules:

**Function 1: check_delivery_responses()**
- Purpose: Check pg_net HTTP responses for recently sent deliveries and update their status
- Query: JOIN webhook_deliveries wd ON net._http_response r WHERE wd.pg_net_request_id = r.id AND wd.status IN ('sent', 'retrying')
- For each matched delivery:
  - If r.status_code >= 200 AND r.status_code < 300: status stays 'sent' (success, no action needed)
  - If r.status_code >= 400 OR r.status_code IS NULL (timeout/error): SET status = 'failed', error_message = COALESCE(r.error_msg, 'HTTP ' || r.status_code::text)
  - If r.timed_out = true: SET status = 'failed', error_message = 'Request timed out'
- Only process deliveries where sent_at > now() - interval '10 minutes' (avoid scanning old records)
- Clean up: DELETE FROM net._http_response WHERE id IN (processed request ids) to prevent the pg_net response table from growing unbounded (pg_net docs recommend this)

**Function 2: process_webhook_retries(p_batch_size integer DEFAULT 10)**
- Purpose: Retry failed deliveries with exponential backoff, jitter, and batch limit
- SELECT from webhook_deliveries WHERE status = 'failed' AND retry_count < 3
- ORDER BY created_at ASC (oldest failures first)
- LIMIT p_batch_size (default 10 per execution, per pitfall research on retry storms)
- For each delivery in the batch:
  1. Check backoff: skip if last_retry_at IS NOT NULL AND last_retry_at > now() - (interval '1 minute' * power(2, retry_count)) (exponential backoff: 1min, 2min, 4min)
  2. Re-fire: SELECT net.http_post(url := target_url, body := payload, headers := '{"Content-Type": "application/json"}'::jsonb) INTO v_new_request_id
  3. UPDATE webhook_deliveries SET pg_net_request_id = v_new_request_id, retry_count = retry_count + 1, status = 'retrying', last_retry_at = now(), error_message = NULL WHERE id = delivery.id
- After the loop: Mark permanently failed: UPDATE webhook_deliveries SET status = 'failed_permanent' WHERE status = 'failed' AND retry_count >= 3
- Log permanent failures to activity_log: INSERT INTO activity_log (event_type, lead_id, broker_id, order_id, details) for each newly failed_permanent delivery with event_type = 'webhook_failed_permanent' and details containing the error_message and retry_count

**pg_cron schedules:**
- SELECT cron.schedule('check-delivery-responses', '30 seconds', 'SELECT check_delivery_responses()');
  NOTE: pg_cron supports sub-minute intervals with the '30 seconds' syntax on Supabase. If this fails, fall back to '* * * * *' (every minute).
- SELECT cron.schedule('retry-failed-webhooks', '*/2 * * * *', 'SELECT process_webhook_retries(10)');

**Important considerations from pitfall research:**
- Batch size of 10 prevents retry storms (pitfall 5 in PITFALLS.md)
- Exponential backoff with natural jitter from pg_cron scheduling interval
- pg_cron max 32 concurrent jobs is not a concern since these are 2 lightweight jobs
- Clean up net._http_response after processing to prevent table bloat
