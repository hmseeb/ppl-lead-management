<objective>
Add timezone-aware contact hours enforcement to the delivery pipeline. When a lead is assigned to a broker who is outside their contact window (wrong hours, weekend paused), the delivery row gets status 'queued' instead of 'pending', preventing the outbound trigger from firing. The queued delivery sits untouched until Phase 11's release job picks it up.

Purpose: Respect broker contact preferences so leads aren't delivered at 2 AM or on weekends when the broker has those settings enabled.
Output: Migration 00016, updated assign_lead() function, updated trigger functions, updated TypeScript types.
</objective>

**Section 2: Create is_within_contact_hours(p_broker_id uuid) RETURNS boolean**
Pure SQL helper function. Logic:
1. Fetch broker's `contact_hours`, `custom_hours_start`, `custom_hours_end`, `weekend_pause`, `timezone` from brokers table
2. If `contact_hours = 'anytime'` or `contact_hours IS NULL`, return TRUE immediately
3. Compute `v_now` as `now() AT TIME ZONE COALESCE(v_timezone, 'America/Los_Angeles')` (default timezone per TZ-01)
4. If `v_weekend_pause = true` AND `EXTRACT(DOW FROM v_now)` is 0 (Sunday) or 6 (Saturday), return FALSE
5. Determine the hour window:
   - If `contact_hours = 'business_hours'`: start=9, end=17 (9 AM to 5 PM)
   - If `contact_hours = 'custom'`: parse `custom_hours_start` and `custom_hours_end` using `EXTRACT(HOUR FROM to_timestamp(v_start_text, 'HH:MI AM')::time)` to get hour integers. Also extract minutes for precision. Compare `v_now::time` against the parsed start and end times.
6. Check if `v_now::time` is between start_time and end_time. Return TRUE if within, FALSE if outside.

Use `to_timestamp('9:00 AM', 'HH:MI AM')::time` pattern to parse the AM/PM format text into a time value. Compare directly with `v_now::time BETWEEN v_start_time AND v_end_time`.

For 'business_hours', use `v_now::time BETWEEN '09:00'::time AND '17:00'::time`.
