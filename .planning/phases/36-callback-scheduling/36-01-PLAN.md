  <action>
Create a new Supabase edge function at `supabase/functions/fire-callback-webhooks/index.ts` using the Deno.serve pattern (same as send-digest and send-alert).

The function does:

1. **Auth check**: Verify `Authorization: Bearer <service_role_key>` header exists (same pattern as send-digest line 189-192). Return 401 if missing.

2. **Create Supabase client**: Use `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` env vars with `createClient` from `https://esm.sh/@supabase/supabase-js@2`.

3. **Query due callbacks**: Select from `callbacks` where `status = 'pending'` AND `scheduled_time <= now()`. Join with `leads` (id, first_name, last_name, email, phone, vertical, credit_score, funding_amount, state) and `brokers` (id, first_name, last_name, email, phone, company_name, crm_webhook_url). These are callbacks whose time has arrived.

4. **Query reminder callbacks**: Select from `callbacks` where `status = 'pending'` AND `scheduled_time > now()` AND `scheduled_time <= now() + interval '20 minutes'`. Same joins. The 20-minute window (not 15) ensures callbacks scheduled 15 min out get caught by the 5-minute cron cycle. To avoid sending duplicate reminders, add a `reminder_sent_at` column check: `reminder_sent_at IS NULL`.

5. **Fire webhooks**: For each due callback, POST to `broker.crm_webhook_url` (skip if null/empty) with payload matching the `callback_created` format from `src/lib/webhooks/callback-webhook.ts`:
```json
{
  "type": "callback_due",
  "callback": { "id", "scheduled_time", "status", "notes", "created_at" },
  "lead": { "id", "first_name", "last_name", "email", "phone", "vertical", "credit_score", "funding_amount", "state" },
  "broker": { "id", "name" (first+last joined), "email", "phone", "company" (company_name) }
}
```
For reminder callbacks, same payload but `type: "callback_reminder"`.

6. **Update status after due webhooks**: After firing callback_due, update the callback's status to `completed` and set `updated_at = now()`.

7. **Update reminder_sent_at**: After firing callback_reminder, update `reminder_sent_at = now()` on the callback row to prevent duplicate reminders.

8. **Error handling**: Use try/catch around each fetch. Log errors with `console.error('fire-callback-webhooks:', type, err)`. Continue processing remaining callbacks even if one webhook fails (fire-and-forget, matching existing pattern). Use 10-second AbortController timeout per request.

9. **Return response**: Return JSON `{ success: true, due_fired: N, reminders_fired: N }` with 200 status.

IMPORTANT: The `reminder_sent_at` column does not exist yet on the callbacks table. Task 2's migration will add it. The edge function must handle the case where this column exists.

Create migration `supabase/migrations/00032_callback_scheduling.sql` with:

1. **Add reminder_sent_at column** to callbacks table:
```sql
ALTER TABLE callbacks ADD COLUMN reminder_sent_at timestamptz;
```
This column tracks whether a reminder webhook was already sent for a callback, preventing duplicates across cron runs.

2. **pg_cron schedule** using the exact same pattern as `00015_daily_digest.sql` (cron.schedule + net.http_post + vault secrets):
```sql
SELECT cron.schedule(
  'fire-callback-webhooks',
  '*/5 * * * *',
  $$
  SELECT net.http_post(
    url := (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'supabase_url')
           || '/functions/v1/fire-callback-webhooks',
    headers := jsonb_build_object(
      'Content-Type', 'application/json',
      'Authorization', 'Bearer ' || (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'service_role_key')
    ),
    body := '{}'::jsonb
  ) AS request_id;
  $$
);
```

Schedule: every 5 minutes (`*/5 * * * *`). This means reminders fire within a 5-minute window of the 15-minute target, which is acceptable precision for a CRM notification.

Add a SQL comment at the top explaining the migration's purpose, the cron schedule, and that vault secrets `supabase_url` and `service_role_key` must already be configured (they are, from Phase 6).

IMPORTANT: Do NOT create a PL/pgSQL function for this. The cron job simply calls the edge function via HTTP. All logic lives in the edge function (Task 1). This matches the daily-digest pattern, not the process_queued_deliveries pattern (which has SQL-level logic).
