  <action>
Create migration file `supabase/migrations/00014_alert_triggers.sql`. Start with the delivery failure trigger.

Create `notify_delivery_failed()` function:
- `RETURNS trigger`, `LANGUAGE plpgsql`, `SECURITY DEFINER`, `SET search_path = public`
- Step 1: `SELECT * INTO v_settings FROM admin_settings LIMIT 1`. Early return if NOT FOUND, or `alert_sms_enabled = false`, or `failure_alert_enabled = false`.
- Step 2: Compute dedup window from `v_settings.dedup_window_minutes`. Check `alert_state` for existing alert where `alert_type = 'delivery_failed'` AND `context_id = NEW.broker_id::text` AND `last_sent_at > now() - v_dedup_window`. If already alerted, UPDATE `suppressed_count + 1` and `last_payload` with current delivery info, then RETURN NEW.
- Step 3: Resolve lead name with NULL-safe concatenation: `TRIM(COALESCE(l.first_name, '') || ' ' || COALESCE(l.last_name, ''))`, fallback to `email`, `phone`, `id::text`. Same pattern for broker name, fallback to `company`, `email`, `id::text`.
- Step 4: Read Vault secrets: `supabase_url` and `service_role_key` from `vault.decrypted_secrets`. Early return if either is NULL.
- Step 5: `PERFORM net.http_post(...)` to `v_supabase_url || '/functions/v1/send-alert'` with JSON body containing: `type = 'delivery_failed'`, `admin_contact_id = v_settings.alert_ghl_contact_id`, `lead_name`, `broker_name`, `channel = NEW.channel`, `error = COALESCE(NEW.error_message, 'Unknown error')`. Headers: `Content-Type: application/json`, `Authorization: Bearer {service_key}`.
- Step 6: UPSERT into `alert_state`: INSERT with `alert_type = 'delivery_failed'`, `context_id = NEW.broker_id::text`, `last_sent_at = now()`, `suppressed_count = 0`, `last_payload` with delivery info. ON CONFLICT `(alert_type, context_id)` DO UPDATE SET `last_sent_at = now()`, `suppressed_count = 0`, `last_payload = EXCLUDED.last_payload`.
- RETURN NEW.

Create trigger definition:
```sql
CREATE TRIGGER trg_alert_delivery_failed
  AFTER UPDATE ON deliveries
  FOR EACH ROW
  WHEN (OLD.status IS DISTINCT FROM 'failed_permanent' AND NEW.status = 'failed_permanent')
  EXECUTE FUNCTION notify_delivery_failed();
```

The WHEN clause is MANDATORY on the trigger definition itself (not inside the function). This prevents the function from firing on every deliveries UPDATE (status changes, retry_count increments, etc.). Only fire when status transitions TO failed_permanent.

Use `PERFORM` (not `SELECT INTO`) for `net.http_post` since we don't need the request ID. This matches the pattern used in existing triggers.

IMPORTANT: Use `PERFORM` not `SELECT ... INTO` for pg_net call. The existing `fire_ghl_delivery()` uses `SELECT net.http_post(...) INTO v_request_id` but we don't need the request_id for alerts. Either pattern works but PERFORM is cleaner.

Reference the research code examples closely. The research has battle-tested the NULL handling, dedup flow, and Vault read patterns.

Create trigger definition:
```sql
CREATE TRIGGER trg_alert_unassigned_lead
  AFTER INSERT ON unassigned_queue
  FOR EACH ROW

```

No WHEN clause needed here because every INSERT into unassigned_queue is a meaningful event worth alerting on. The dedup check inside the function handles repeat entries for the same lead.

- No broker name resolution needed
- Payload includes `reason` field (not `broker_name`, `channel`, `error`)
