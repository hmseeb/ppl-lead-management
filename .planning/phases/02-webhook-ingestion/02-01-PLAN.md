**Step 1: Create src/lib/webhooks/schemas.ts**

Define Zod schema `incomingLeadSchema` matching the GHL payload and leads table columns:
```
z.object({
  first_name: z.string().min(1).optional(),
  last_name: z.string().min(1).optional(),
  email: z.string().email().optional().or(z.literal('')),
  phone: z.string().min(7).optional().or(z.literal('')),
  business_name: z.string().optional(),
  vertical: z.string().min(1),
  credit_score: z.coerce.number().int().min(300).max(850).optional(),
  funding_amount: z.coerce.number().positive().optional(),
  funding_purpose: z.string().optional(),
  state: z.string().optional(),
  ai_call_notes: z.string().optional(),
  ai_call_status: z.string().optional(),
  ghl_contact_id: z.string().min(1),
})
```

Key decisions:
- `vertical` and `ghl_contact_id` are required (min 1 char). Everything else optional because GHL payloads are inconsistent.
- Use `z.coerce.number()` for credit_score and funding_amount because GHL may send them as strings.
- Allow empty string for email/phone with `.or(z.literal(''))` since GHL sometimes sends empty strings instead of omitting.
- Export the inferred type as `IncomingLeadPayload`.

**Step 2: Create src/app/api/leads/incoming/route.ts**

Export an async `POST` function that:

1. Parse request body with `request.json()`. Wrap in try/catch. If JSON parsing fails, return 400 `{ error: 'invalid_json' }`.

2. Validate with `incomingLeadSchema.safeParse(body)`. If invalid, return 400 with `{ error: 'invalid_payload', details: result.error.flatten() }`. This satisfies HOOK-03.

3. Store the raw payload: set `raw_payload = body` (the original unparsed body, for debugging GHL changes).

4. **Idempotency check (HOOK-05):** Use Supabase upsert with `onConflict: 'ghl_contact_id'`. If the lead already exists (conflict on ghl_contact_id UNIQUE constraint), update only fields that are non-null in the new payload (merge, don't overwrite with nulls). Return `{ lead_id, status: 'duplicate', message: 'lead already exists' }` with status 200 (not an error, just idempotent). To detect if it was an insert vs update, check if the returned `created_at` is within the last 2 seconds. Alternatively, use a simpler approach: first try `.select('id').eq('ghl_contact_id', data.ghl_contact_id).maybeSingle()`. If found, return 200 with existing lead_id and skip assignment. If not found, insert.

   Simpler approach (preferred): SELECT first by ghl_contact_id. If exists, return 200 with existing ID (true idempotency, no re-assignment). If not exists, INSERT.

5. Insert lead using `createAdminClient()` (service role bypasses RLS). Insert with all validated fields plus `raw_payload: body, status: 'pending'`. Select the `id` back.

6. Call `assignLead(lead.id)` from `@/lib/assignment/assign.ts`. This triggers the Postgres function.

7. Return `NextResponse.json({ lead_id, assignment }, { status: 200 })`.

CRITICAL per pitfalls research: Do NOT add any slow operations between receiving the request and returning 200. The assignment engine is a single Postgres RPC call, which is fast. No outbound HTTP calls in this handler (delivery is handled by Phase 3). Total should be well under 2 seconds.

Do NOT add webhook secret verification yet. GHL Custom Webhooks' secret mechanism varies. We'll add it when we test with real GHL webhooks.
