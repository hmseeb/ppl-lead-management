<objective>
Enforce strict data isolation for the broker portal at both application and database levels.

Purpose: Prevent any broker from accessing another broker's leads, orders, or deliveries. Application-level filtering is the primary mechanism (since portal uses server-side code with createAdminClient). RLS policies are defense-in-depth for any future client-side access.
Output: RLS migration, broker-scoped query module, mutation ownership guard
</objective>

<task type="auto">
  <name>Task 1: RLS policies for broker data isolation</name>
  <files>
    supabase/migrations/00027_broker_rls_policies.sql
  </files>
  <action>
    Create migration that replaces the permissive "allow all anon reads" policies with broker-scoped ones.

    For each table:

    1. **leads** table:
       - DROP existing "Allow anon read access to leads" policy
       - CREATE POLICY "Broker can read own leads" ON leads FOR SELECT TO anon
         USING (assigned_broker_id = current_setting('request.headers')::json->>'x-broker-id')
       - Keep service role bypass (implicit, service role always bypasses RLS)

    2. **orders** table:
       - DROP existing "Allow anon read access to orders" policy
       - CREATE POLICY "Broker can read own orders" ON orders FOR SELECT TO anon
         USING (broker_id = current_setting('request.headers')::json->>'x-broker-id')

    3. **deliveries** table:
       - DROP existing "Allow anon read access to deliveries" policy
       - CREATE POLICY "Broker can read own deliveries" ON deliveries FOR SELECT TO anon
         USING (broker_id = current_setting('request.headers')::json->>'x-broker-id')

    Note: brokers table keeps its existing "Allow anon read access to brokers" policy since ppl-onboarding needs it.
    Note: activity_log and unassigned_queue keep existing policies since they are admin-only tables not exposed to portal.
  </action>

  </files>
  <action>
    Create `src/lib/portal/queries.ts` with broker-scoped query helpers.
    These use createAdminClient() (service role) but ALWAYS filter by brokerId.
    This is the primary isolation mechanism. RLS is backup.

    Exports:
    - getPortalBroker(brokerId: string) - fetches broker profile for the authenticated broker
    - getPortalLeads(brokerId: string, opts?: { limit?: number, offset?: number, status?: string }) - fetches leads assigned to this broker
    - getPortalOrders(brokerId: string) - fetches orders belonging to this broker
    - getPortalDeliveries(brokerId: string, opts?: { leadId?: string }) - fetches deliveries for this broker

    Each function:
    1. Creates admin client
    2. Builds query with .eq('assigned_broker_id', brokerId) or .eq('broker_id', brokerId)
    3. Returns typed data

    This centralizes all portal data access so no portal code can accidentally query without the broker filter.
  </action>
