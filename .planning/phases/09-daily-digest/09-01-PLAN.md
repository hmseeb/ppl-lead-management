Create migration file with three components:

**1. digest_runs tracking table:**
```sql
CREATE TABLE digest_runs (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  run_at timestamptz NOT NULL DEFAULT now(),
  period_start timestamptz NOT NULL,
  period_end timestamptz NOT NULL,
  status text NOT NULL DEFAULT 'sent' CHECK (status IN ('sent', 'failed')),
  stats jsonb,
  error_message text,
  created_at timestamptz NOT NULL DEFAULT now()
);
```
- Enable RLS with anon read-only policy (matches admin_settings pattern from migration 00012)
- Add comment explaining the table's purpose

**2. Performance index on deliveries.created_at:**
```sql
CREATE INDEX IF NOT EXISTS idx_deliveries_created_at ON deliveries(created_at);
```
This prevents sequential scans on time-bounded queries used by both the digest and the dashboard's fetchDeliveryStats().

**3. pg_cron schedule using pg_net to call send-digest edge function:**
Use the exact pattern from migration 00014 (notify_delivery_failed) for Vault secret reading:
```sql
SELECT cron.schedule(
  'daily-digest',
  '0 16 * * *',
  $$
  SELECT net.http_post(
    url := (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'supabase_url')
           || '/functions/v1/send-digest',
    headers := jsonb_build_object(
      'Content-Type', 'application/json',
      'Authorization', 'Bearer ' || (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'service_role_key')
    ),
    body := '{}'::jsonb
  ) AS request_id;
  $$
);
```

Schedule fires at 16:00 UTC = 8 AM PST (9 AM during PDT). Accept the 1-hour DST drift per project decision in STATE.md.

Add a header comment block documenting the schedule, the DST behavior, and the Vault secrets required.

<task type="auto">
  <name>Task 2: Create send-digest edge function with stats queries, HTML email, and SMS</name>
  <files>supabase/functions/send-digest/index.ts</files>
  <action>
Create the send-digest edge function following the exact patterns from deliver-ghl/index.ts and send-alert/index.ts.

**Structure:**

1. **Auth check:** Verify Bearer token in Authorization header (same pattern as send-alert/index.ts lines 4-6).

2. **Supabase client:** Create client using `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` env vars (same pattern as deliver-ghl/index.ts lines 82-84).

3. **Get admin contact ID:** Query `admin_settings` table for `alert_ghl_contact_id` field. If not found, return 500 with clear error. Do NOT hardcode the contact ID.

4. **Determine time period:** Query `digest_runs` table for the most recent successful run's `period_end`. If no prior run exists, default to `now() - 24 hours`. Set `periodEnd` to `now()`.

5. **Query stats (12 parallel counts):** Replicate the exact query pattern from `fetchDeliveryStats()` in `src/lib/queries/dashboard.ts` but use `periodStart`/`periodEnd` bounds instead of `todayStart`. Use `Promise.all()` for all 12 queries:
   - leads received (created_at in period)
   - leads assigned (created_at in period, status = 'assigned')
   - leads unassigned (created_at in period, status = 'unassigned')
   - total deliveries (created_at in period)
   - sent deliveries (created_at in period, status = 'sent')
   - failed deliveries (created_at in period, status IN ['failed', 'failed_permanent'])
   - webhook total (channel = 'crm_webhook')
   - webhook failed (channel = 'crm_webhook', status IN ['failed', 'failed_permanent'])
   - email total (channel = 'email')
   - email failed (channel = 'email', status IN ['failed', 'failed_permanent'])
   - sms total (channel = 'sms')
   - sms failed (channel = 'sms', status IN ['failed', 'failed_permanent'])

   Use `.gte('created_at', periodStart).lte('created_at', periodEnd)` for all queries.

6. **Build HTML email body (buildDigestEmailHtml function):**
   Table-based layout with ALL inline styles (no CSS classes). Use the exact template structure from 09-RESEARCH.md Pattern 4:
   - Blue header (#1a73e8) with "PPL Lead Management - Daily Digest" and formatted date
   - Lead Activity section: received, assigned, unassigned (red text for unassigned if > 0)
   - Deliveries section: total, sent, failed (red text for failed if > 0)
   - By Channel section: webhook sent/failed, email sent/failed, sms sent/failed
   - Footer: "Delivered by PPL Lead Management"
   - Use zebra-striped rows (alternating #f8f9fa background)
   - Max-width 600px, Arial/Helvetica/sans-serif font stack
   - Format date as readable string (e.g., "March 13, 2026")

7. **Build SMS body (buildDigestSmsBody function):**
   Compact multi-line format:
   ```
   Daily Digest - Mar 13
   Leads: 42 received, 38 assigned, 4 unassigned
   Deliveries: 52 sent, 4 failed
   Webhook: 20/1 | Email: 18/2 | SMS: 14/1
   ```
   Where channel format is `sent/failed`.

8. **Send email via GHL:**
   CRITICAL: Use `html` field for email body, NOT `message`. The `message` field causes blank emails. This was a real bug in this codebase.
   ```typescript
   {
     type: 'Email',
     contactId: adminContactId,
     subject: `Daily Digest - ${formattedDate}`,
     html: emailHtml,        // <-- HTML goes here, NOT 'message'
     emailFrom: fromEmail,   // GHL_FROM_EMAIL env var or 'leads@pplleads.com'
   }
   ```
   Use GHL base URL `https://services.leadconnectorhq.com/conversations/messages`, Version header `2021-07-28`, Bearer token from `GHL_API_TOKEN` env var.

9. **Send SMS via GHL:**
   ```typescript
   {
     type: 'SMS',
     contactId: adminContactId,
     message: smsBody,       // <-- Plain text for SMS
   }
   ```

10. **Record digest run:** Insert into `digest_runs` with period_start, period_end, status ('sent' or 'failed'), stats jsonb snapshot, and error_message if applicable. Record AFTER both sends attempt. If email succeeds but SMS fails (or vice versa), still record as 'sent' with a note in stats about partial failure.

11. **Error handling:**
    - If GHL returns 429, log the error and record as 'failed' in digest_runs. Do NOT retry (per research recommendation).
    - If GHL returns other errors, log and record as 'failed'.
    - Always return a response (200 for success, 500/502 for failure) so pg_net doesn't hang.

**Constants at top of file:**
```typescript
const GHL_BASE_URL = 'https://services.leadconnectorhq.com'
const GHL_API_VERSION = '2021-07-28'
```

**Import:** `import { createClient } from 'https://esm.sh/@supabase/supabase-js@2'`

Do NOT import date-fns or any other library. Use native Date methods and Intl.DateTimeFormat for date formatting in the edge function (keeps it lightweight, avoids esm.sh resolution issues).
