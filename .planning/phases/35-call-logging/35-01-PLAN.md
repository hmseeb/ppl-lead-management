  <action>
Create migration file `supabase/migrations/00031_call_logs.sql`:

```sql
CREATE TABLE call_logs (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  lead_id uuid NOT NULL REFERENCES leads(id),
  broker_id uuid NOT NULL REFERENCES brokers(id),
  outcome text NOT NULL CHECK (outcome IN ('transferred', 'callback_booked', 'no_answer', 'voicemail')),
  duration integer NOT NULL DEFAULT 0,
  retell_call_id text NOT NULL,
  notes text,
  created_at timestamptz NOT NULL DEFAULT now()
);
```

Indexes to add (needed by Phase 37 reporting dashboard):
- `idx_call_logs_broker_id` on `(broker_id)` for broker-scoped queries
- `idx_call_logs_created_at` on `(created_at)` for date range filtering
- Composite index `idx_call_logs_broker_created` on `(broker_id, created_at)` for the combined broker + date range query pattern Phase 37 will use

No updated_at column or trigger needed. Call logs are immutable (write-once, never updated).

Add call_logs table types to `src/lib/types/database.ts` inside `Database.public.Tables`, following the exact same structure as the callbacks entry. Fields:
- Row: id (string), lead_id (string), broker_id (string), outcome (string), duration (number), retell_call_id (string), notes (string | null), created_at (string)
- Insert: id optional, lead_id required, broker_id required, outcome required, duration optional (defaults 0), retell_call_id required, notes optional/null, created_at optional
- Update: all optional
- Relationships: lead_id -> leads(id), broker_id -> brokers(id)

Place the `call_logs` entry alphabetically after `callbacks` in the Tables object.

Create `src/app/api/call-logs/route.ts` following the exact pattern from `src/app/api/callbacks/route.ts`:

1. Import NextResponse, z, createAdminClient from the same paths as callbacks/route.ts.

2. Define Zod schema `createCallLogSchema`:
   - lead_id: z.string().uuid()
   - broker_id: z.string().uuid()
   - outcome: z.enum(['transferred', 'callback_booked', 'no_answer', 'voicemail'])
   - duration: z.number().int().min(0).default(0)
   - retell_call_id: z.string().min(1)
   - notes: z.string().optional()

3. POST handler:
   - Parse JSON body with try/catch (return 400 on invalid JSON, matching callbacks pattern)
   - Validate with safeParse, return 400 with flattened errors on failure
   - Verify lead exists (select id from leads where id = lead_id, return 404 if not found)
   - Verify broker exists (select id from brokers where id = broker_id, return 404 if not found)
   - Insert into call_logs table, select back all columns, return 201 with the record
   - On insert failure return 500 with error message

4. Log to activity_log with event_type 'call_logged', including lead_id, broker_id, and details: { call_log_id, outcome, retell_call_id }.

No webhook firing needed (unlike callbacks). This is a pure data ingestion endpoint.
