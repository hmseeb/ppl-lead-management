<objective>
Create an auto-reassignment system that processes unassigned queue leads when an order is activated or unpaused.

Purpose: Eliminate admin intervention for matching unassigned leads to newly available orders.
Output: reassignUnassignedLeads() function wired into order status changes, with full scoring, delivery, and activity logging.
</objective>

  <action>
**1. Create src/lib/assignment/reassign.ts:**

New module that:
- Queries `unassigned_queue` for all entries where `resolved = false`
- Joins with `leads` table to get lead data (credit_score, funding_amount, vertical)
- For each unresolved lead, calls `assignLead(lead_id)` from `@/lib/assignment/assign`
- If assignment returns `status: 'assigned'`:
  - Marks the queue entry as resolved (resolved=true, resolved_at=now)
  - Logs to activity_log with event_type 'auto_reassignment'
  - Dispatches delivery via dispatchDelivery()
- If assignment returns `status: 'unassigned'`, skips (lead stays in queue)
- Wraps each lead in try/catch so one failure doesn't block others
- Returns a summary: { processed: number, assigned: number, failed: number }

**2. Modify src/lib/actions/orders.ts:**

In `updateOrderStatus()`, after the status update succeeds and the new status is 'active' (meaning the order was activated or unpaused), call `reassignUnassignedLeads()` as fire-and-forget (don't block the response).

Also call it after `createOrder()` succeeds, since a new active order could match queued leads.
