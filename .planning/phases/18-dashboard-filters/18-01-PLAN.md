Refactor all 4 query functions in `src/lib/queries/dashboard.ts` to accept an optional `DashboardFilters` parameter and apply filtering. Import `DashboardFilters` and `getDateRange` from `@/lib/types/dashboard-filters`.

**fetchKpis(filters?: DashboardFilters):**
- Use `getDateRange(filters ?? {})` to get `{ from, to }` date range
- Replace all hardcoded `startOfDay(now)` / `weekStart` / `monthStart` with the filter range:
  - `leadsToday` (rename concept to "leads in range"): `.gte('created_at', from).lte('created_at', to)`
  - Remove `leadsThisWeek` and `leadsThisMonth` separate queries. They only make sense for the "today" default. Instead, just return the single count for the selected range. Keep the return shape compatible: set `leadsThisWeek` and `leadsThisMonth` to 0 when filters are active (non-today), or compute them normally when `date_preset` is 'today' or undefined with no custom dates.
  - Actually, simpler approach: ALWAYS compute all three (leadsToday, leadsWeek, leadsMonth) using their original logic. The date filter only affects leadsToday (the primary KPI). The subtitle "X this week / Y this month" remains useful context regardless of filter. So: only apply the filter date range to `leadsToday`, `assigned` (with `.gte('created_at', from)`), `rejectedToday`, `failedDeliveries`, `failedPermanentDeliveries`. Keep `leadsWeek` and `leadsMonth` as-is for context.
  - Wait, re-reading the requirements: "all KPI cards and delivery stats update to reflect that range". So ALL date-scoped queries must use the filter range. The "this week / this month" subtitle becomes less useful but the requirement is clear.
  - Final approach: Replace ALL date-scoped `.gte('created_at', todayStart)` with `.gte('created_at', from).lte('created_at', to)`. Remove weekStart/monthStart queries entirely. Update the return type to not include leadsThisWeek/leadsThisMonth (they are only used in subtitle text, which we'll update in Plan 18-02 or handle here by returning the date range label instead).
  - Actually, keep the return shape stable to minimize downstream changes. Set `leadsThisWeek: 0` and `leadsThisMonth: 0` (the KPI card subtitle will need updating but that's a minor display change we can handle in the component). Remove those two parallel queries.
- For broker filter: if `filters.broker_id` is set, add `.eq('assigned_broker_id', filters.broker_id)` to lead queries, `.eq('broker_id', filters.broker_id)` to delivery queries. For `activeBrokers` and `activeOrders`, filter by `.eq('id', filters.broker_id)` and `.eq('broker_id', filters.broker_id)` respectively.
- For vertical filter: if `filters.vertical` is set, add `.eq('vertical', filters.vertical)` to lead queries. For delivery queries, we need a join. Since deliveries don't have vertical directly, use `.eq('leads.vertical', filters.vertical)` with a join. Actually, Supabase count queries with head:true can't do joins easily. Alternative: for delivery queries with vertical filter, do a subquery approach: first get lead IDs with that vertical, then filter deliveries. This is complex. Simpler: skip vertical filtering on delivery-only queries (deliveries inherently relate to leads which have verticals, but the count queries use head:true). Best approach: use `.in('lead_id', subquery)` but Supabase JS doesn't support subqueries directly. Most pragmatic: when vertical filter is active, fetch lead IDs with that vertical first (single query), then use `.in('lead_id', leadIds)` on delivery queries. If there are too many lead IDs, this won't scale, but for an admin dashboard with moderate data it's fine.
  - Even simpler: for delivery queries when vertical is set, select deliveries joined with leads and filter: `supabase.from('deliveries').select('id, leads!inner(vertical)', { count: 'exact', head: true }).eq('leads.vertical', filters.vertical)`. The `!inner` join with `head: true` and `count: 'exact'` should work for counting. Test this pattern.
- For `activeOrders` with vertical filter: use `.contains('verticals', [filters.vertical])` since orders have a `verticals` array field.
- For `activeBrokers` with vertical filter: use `.eq('primary_vertical', filters.vertical)` since brokers have a `primary_vertical` field.

**fetchDeliveryStats(filters?: DashboardFilters):**
- Same pattern: use `getDateRange()` for date filtering, replace all `todayStart` with `from`/`to` range
- Apply broker filter with `.eq('broker_id', filters.broker_id)` on delivery queries and `.eq('assigned_broker_id', filters.broker_id)` on lead queries
- Apply vertical filter: on lead queries use `.eq('vertical', filters.vertical)`. On delivery queries, use the inner join pattern: `.select('id, leads!inner(vertical)', { count: 'exact', head: true }).eq('leads.vertical', filters.vertical)` when vertical is set, otherwise use original select
- Build query chains conditionally: create base query, then chain `.eq()` calls only if filter values exist

**fetchLeadVolume7Days(filters?: DashboardFilters):**
- Rename to `fetchLeadVolume(filters?: DashboardFilters)` (it's no longer always 7 days)
- Use `getDateRange()` to determine the full range
- Loop from `from` to `to` date, one day at a time, counting leads per day (same pattern as current)
- Apply broker filter: `.eq('assigned_broker_id', filters.broker_id)` if set
- Apply vertical filter: `.eq('vertical', filters.vertical)` if set
- Keep the same return shape: `{ date, label, count }[]`
- For the label, use `format(day, 'MMM d')` instead of `format(day, 'EEE')` when range is > 7 days
