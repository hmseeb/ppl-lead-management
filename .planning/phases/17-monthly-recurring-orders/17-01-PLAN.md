  <action>
Create migration `supabase/migrations/00023_monthly_reset.sql` with:

1. **`reset_monthly_orders()` function** (plpgsql, RETURNS integer):
   - Use a FOR loop over all orders WHERE `order_type = 'monthly'` AND `status IN ('active', 'completed')` AND `leads_delivered > 0`
   - For each order:
     a. Store `v_prev_delivered := order.leads_delivered` for audit logging
     b. UPDATE the order: SET `leads_delivered = 0`, `leads_remaining = total_leads`, `status = 'active'`, `updated_at = now()`
     c. INSERT INTO `activity_log` with `event_type = 'monthly_reset'`, `order_id = order.id`, `broker_id = order.broker_id`, and `details = jsonb_build_object('previous_delivered', v_prev_delivered, 'total_leads', order.total_leads, 'reset_at', now()::text)`
     d. Increment a counter
   - RETURN the counter (number of orders reset)
   - Add COMMENT ON FUNCTION explaining purpose

2. **pg_cron schedule**:
   - Schedule name: `'monthly-order-reset'`
   - Cron expression: `'0 0 1 * *'` (midnight UTC on the 1st of each month)
   - SQL body: `'SELECT reset_monthly_orders()'`
   - Pattern matches existing pg_cron schedules in migrations 00015 and 00017

Important: The function must ONLY touch orders with `order_type = 'monthly'`. One-time orders (`order_type = 'one_time'`) must never be selected. The WHERE clause is the safety guard for success criteria 4.

Important: Orders with `leads_delivered = 0` should be skipped (no-op, no log entry). Only reset orders that actually received leads during the previous month.
